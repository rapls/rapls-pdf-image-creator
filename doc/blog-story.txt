<h1>WordPressプラグインを開発してWordPress.orgに公開するまでの全記録</h1>

<strong>「PDFをアップロードしたら、自動でサムネイルが生成されたらいいのに」</strong>

そんな素朴な思いから始まったプラグイン開発。完成までに何度もWordPress.orgの審査に落ち、コードを書き直し、セキュリティを見直し...。この記事では、<strong>Rapls PDF Image Creator</strong>というプラグインを開発し、公式ディレクトリに公開するまでの全工程を、失敗談も含めて赤裸々に綴ります。

<!--more-->

<h2>きっかけ：PDFのサムネイルが欲しい</h2>

私のサイトでは、PDFファイルを頻繁にアップロードします。カタログ、マニュアル、レポート...。

でも、WordPressのメディアライブラリを開くと、PDFはすべて同じアイコン。

<!-- 📸 スクリーンショット1: メディアライブラリでPDFが同じアイコンで並んでいる様子 -->

「このPDF、何だっけ？」

いちいちダウンロードして開かないと中身がわからない。これが地味にストレスでした。

既存のプラグインも探しましたが、どれも微妙に違う。GhostScriptが必要だったり、有料だったり、更新が止まっていたり。

<strong>「ないなら作ろう」</strong>

エンジニアなら誰もが一度は思うこのセリフ。私もその誘惑に負けました。

<h2>開発環境の準備</h2>

まずはローカル環境の構築から。私はLocal by Flywheelを使っています。

<pre><code>必要なもの：
- Local by Flywheel（ローカルWordPress環境）
- VS Code（エディタ）
- PHP 7.4以上
- ImageMagick（PDF変換用）</code></pre>

プラグインの基本構造はこんな感じで設計しました：

<pre><code>rapls-pdf-image-creator/
├── rapls-pdf-image-creator.php  # メインファイル
├── includes/
│   ├── Plugin.php               # シングルトンのエントリーポイント
│   ├── Generator.php            # PDF→画像変換のコア
│   ├── Settings.php             # 設定値の管理
│   ├── Admin.php                # 管理画面
│   ├── MediaLibrary.php         # メディアライブラリ連携
│   └── Engine/
│       └── ImagickEngine.php    # ImageMagick実装
├── admin/
│   ├── views/                   # 設定画面のテンプレート
│   ├── css/                     # 管理画面用CSS
│   └── js/                      # 管理画面用JS
└── languages/                   # 翻訳ファイル</code></pre>

名前空間（Namespace）を使ってクラスを整理。オートローダーも自前で実装しました。

<h2>最初の壁：PDF変換エンジン</h2>

PDFを画像に変換する方法は主に2つあります：

<ol>
<li><strong>ImageMagick（Imagick PHP拡張）</strong> - PHPから直接呼び出せる</li>
<li><strong>GhostScript</strong> - コマンドラインツール、exec()で実行</li>
</ol>

最初は両方に対応しようと思っていました。「ユーザーの環境に合わせて選べるようにしよう」と。

でも、これが後で大きな問題になるとは、この時は知る由もありませんでした...。

<h2>機能実装：こだわったポイント</h2>

<h3>1. アップロード時の自動生成</h3>

PDFをアップロードしたら、何もしなくてもサムネイルが生成される。これが一番こだわったポイントです。

<code>add_attachment</code>フックを使って、PDFがアップロードされたタイミングを検知：

<pre><code>add_action('add_attachment', function($attachmentId) {
    $mime = get_post_mime_type($attachmentId);
    if ($mime === 'application/pdf') {
        // サムネイル生成処理
    }
});</code></pre>

<!-- 📸 スクリーンショット2: PDFアップロード後、自動でサムネイルが生成された様子 -->

<h3>2. アイキャッチ画像として設定</h3>

生成したサムネイルをPDFの「アイキャッチ画像」として設定できるようにしました。

これで<code>get_post_thumbnail_id()</code>がPDF添付ファイルでも使えるようになります。テーマ開発者にとっては嬉しい機能のはず。

<h3>3. 既存PDFの一括処理</h3>

プラグインを有効化する前にアップロードしたPDFはどうする？

「一括生成」機能を実装しました。AJAXで1件ずつ処理して、プログレスバーで進捗を表示。

<h2>WordPress.org審査：地獄の始まり</h2>

プラグインが完成し、意気揚々とWordPress.orgに提出。

数日後、メールが届きました。

<blockquote>
<strong>Your plugin has been reviewed and we have concerns...</strong>
</blockquote>

<strong>却下。</strong>

ここから、審査との長い戦いが始まりました。

<h3>指摘1: プラグイン名が一般的すぎる</h3>

最初は「PDF Image Creator」という名前でした。

<blockquote>
プラグイン名にユニークなプレフィックスを付けてください。
</blockquote>

「Rapls PDF Image Creator」に変更。Raplsは私のブランド名です。

<h3>指摘2: exec()は使用禁止</h3>

これが一番痛かった。

GhostScriptエンジンは<code>exec()</code>関数でコマンドを実行していました。

<blockquote>
exec()、shell_exec()、system()などのシェル実行関数は、セキュリティ上の理由から使用できません。
</blockquote>

せっかく実装したGhostScript対応を、泣く泣く削除。ImageMagick一本に絞りました。

<h3>指摘3: プレフィックスの統一</h3>

関数名、定数、オプション名...すべてに一貫したプレフィックスが必要でした。

<pre><code>// 修正前
function get_pdf_thumbnail() { ... }
define('PLUGIN_VERSION', '1.0.0');

// 修正後
function rapls_pic_get_thumbnail() { ... }
define('RAPLS_PIC_VERSION', '1.0.0');</code></pre>

<h3>指摘4: flush_rewrite_rules()の削除</h3>

有効化・無効化時に<code>flush_rewrite_rules()</code>を呼んでいましたが、これも指摘を受けました。

<blockquote>
このプラグインはカスタム投稿タイプやリライトルールを登録していません。不要な処理は削除してください。
</blockquote>

確かに、意味もなく入れていました。反省。

<h3>指摘5: error_log()のファイルパス露出</h3>

デバッグ用に入れていた<code>error_log()</code>が問題に。

<pre><code>// 修正前（危険）
error_log('Error in ' . __FILE__ . ': ' . $e->getMessage());

// 修正後（安全）
if (defined('WP_DEBUG') && WP_DEBUG) {
    error_log('Rapls PDF Image Creator: ' . $e->getMessage());
}</code></pre>

サーバーのファイルパスが露出するのはセキュリティリスク。WP_DEBUGが有効な時だけログを出すように修正しました。

<h3>指摘6: load_plugin_textdomain()は不要</h3>

翻訳の読み込み処理を入れていましたが...

<blockquote>
WordPress 4.6以降、WordPress.orgでホストされるプラグインは翻訳が自動的に読み込まれます。load_plugin_textdomain()は非推奨です。
</blockquote>

知らなかった...。削除しました。

<h3>指摘7: 日本語翻訳のスタイル</h3>

日本語翻訳ファイルも審査対象でした。

WordPress日本語チームの<a href="https://ja.wordpress.org/team/handbook/translation/translation-style-guide/">翻訳スタイルガイド</a>に従う必要があります。

<pre><code>// 修正前
"PDFファイルからサムネイルを生成"

// 修正後（半角英字の前後にスペース）
"PDF ファイルからサムネイルを生成"</code></pre>

細かいですが、約50箇所を修正しました。

<h2>最終的なコード品質</h2>

審査を通過するために、以下の点を徹底しました：

<h3>セキュリティ</h3>
<ul>
<li>すべての入力値をサニタイズ（<code>sanitize_text_field()</code>、<code>absint()</code>など）</li>
<li>すべての出力をエスケープ（<code>esc_html()</code>、<code>esc_attr()</code>、<code>esc_url()</code>）</li>
<li>AJAX処理には必ずnonce検証と権限チェック</li>
<li>カスタムHTMLには<code>wp_kses_post()</code>でサニタイズ</li>
</ul>

<h3>コーディング規約</h3>
<ul>
<li>WordPress Coding Standardsに準拠</li>
<li>PHPDoc形式のコメント</li>
<li>適切な名前空間とクラス設計</li>
</ul>

<h3>互換性</h3>
<ul>
<li>PHP 7.4以上で動作（<code>readonly</code>プロパティや<code>match</code>式は使わない）</li>
<li>WordPress 5.0以上で動作</li>
</ul>

<h2>公開後の反省と学び</h2>

<h3>良かったこと</h3>
<ul>
<li>審査プロセスを通じて、セキュリティ意識が大幅に向上した</li>
<li>WordPressのコーディング規約を深く理解できた</li>
<li>「動けばいい」から「正しく動く」へ意識が変わった</li>
</ul>

<h3>反省点</h3>
<ul>
<li>最初からWordPress.orgのガイドラインを読んでおくべきだった</li>
<li>GhostScript対応に時間をかけすぎた（結局削除）</li>
<li>翻訳スタイルガイドの存在を知らなかった</li>
</ul>

<h3>これからプラグインを開発する人へ</h3>

<ol>
<li><strong>まずガイドラインを読む</strong> - <a href="https://developer.wordpress.org/plugins/wordpress-org/detailed-plugin-guidelines/">Plugin Guidelines</a>は必読</li>
<li><strong>Plugin Check プラグインを使う</strong> - 提出前に自動チェックできる</li>
<li><strong>exec()系は諦める</strong> - 外部コマンド実行は審査に通らない</li>
<li><strong>プレフィックスは最初から統一</strong> - 後から変更するのは大変</li>
<li><strong>セキュリティは妥協しない</strong> - サニタイズ・エスケープ・nonce検証</li>
</ol>

<pre><code>&lt;?php
